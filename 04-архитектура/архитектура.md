# Архитектура SmartVPT-бота

## Стек
- **Язык:** Python (Лёва).
- **Telegram:** Telegram Bot API.
- **MAX:** MAX Bot API.
- **Хранение:** на стороне сервера клиента (по серверу на клуб).
- **Источник лидов:** Google Sheets — менеджер клуба заносит туда, бот импортирует.
- **Хостинг:** NetAngels VPS, **по отдельному серверу на каждого клиента** (см. [[adr-002-vps-per-client]]).
- **Деплой:** ручной на VPS, `systemd` юнит на каждом сервере.
- **Логи:** `journalctl` + файл, ротация.

## Компоненты
```
[Менеджер клуба] --заносит лида--> [Google Sheets клиента]
                                          |
                                          v
                              [VPT-бот процесс на VPS клиента]
                                          |
              +-----------+----------------+-------------------+
              v           v                v                   v
    [Telegram Bot API]  [MAX Bot API]  [локальная БД]   [journalctl/файл]
              |
              v
    [Координатор / Тренер / Владелец]
```

## Многотенантность
- **Один сервер = один клиент.** Никаких shared-баз между клубами.
- **Отдельный токен бота** в BotFather на клиента (юрлицо клиента, не наше — переносит юр. риск ПД).
- **Отдельный домен / поддомен** при необходимости.
- При новом клиенте — новый VPS NetAngels + деплой по [[runbooks/новый-клиент|runbook]].

## Известные ограничения
- **Лимит страницы Telegram** — длинные списки клиентов разбиваются на несколько страниц (баг #5 закрыт 27.04).
- **Кнопки в TG** — раздувать UI нельзя, превращается в приложение, ломает позиционирование (5–7к/мес продукт).
- **Запросы на «выпадающие списки»** — отклоняем; пишем в комментариях.

## Безопасность и юридика
- Персональные данные клиентов клуба (ФИО, телефоны) идут через мессенджер клиента (его юрлицо в МАХ). Юр. риск — на клиенте.
- **Не показываем клиентам** карту-схему архитектуры — копируют.
- Подробности юр-рисков: `smartvpt-vault/05-юр/юр-риски.md`.
