# Runbook: деплой фикса на сервер клиента

## Контекст
- Один VPS = один клиент.
- Деплой ручной: `git pull` + перезапуск `systemd`-юнита.
- Бэкап перед деплоем — обязателен (сейчас руками; CI/CD — в бэклоге).

## Стандартная процедура

```bash
# 1. SSH на сервер клиента
ssh <user>@<vps-ip>

# 2. Бэкап локальной БД
cp /path/to/bot.db /path/to/backups/bot-$(date +%Y%m%d-%H%M).db

# 3. Подтянуть код
cd /path/to/vptbot
git pull origin main

# 4. (если меняется зависимость) обновить venv
source .venv/bin/activate
pip install -r requirements.txt

# 5. Перезапуск
sudo systemctl restart vptbot

# 6. Проверка
sudo systemctl status vptbot
sudo journalctl -u vptbot -n 100 --no-pager
```

## Откат
Если фикс ломает бот:
```bash
git log --oneline -10               # найти предыдущий коммит
git checkout <prev-sha>             # откат кода
sudo systemctl restart vptbot
# при необходимости — восстановить БД из бэкапа:
cp /path/to/backups/bot-<timestamp>.db /path/to/bot.db
```

## Когда катить на ВСЕХ клиентов
- Только если фикс универсальный и проверен хотя бы на одном клиенте.
- Порядок: сначала Дублёр (Илья поблизости, быстро поймает регресс), потом Барнаул.
- В каждый клуб — отдельный SSH, отдельный `git pull`, отдельный рестарт.

## NB
- Не «катаем» одним скриптом на все серверы — ловим регрессы поштучно.
- При раскатке нескольких фиксов подряд — **между раскатками выждать рабочий день**, чтобы поймать жалобы.
